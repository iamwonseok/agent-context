# Scenario 006: Manager Link Best Practices

## Context
매니저 'wonseok'은 월간 프로젝트 리뷰를 진행 중입니다. 현재 '백엔드 API 개선' 에픽이 진행 중이며, 여러 팀 간의 의존성이 복잡하게 얽혀 있는 상황입니다.

## Step 1: 의존성 병목 식별 (Blocks / Is blocked by)
매니저가 에픽의 상태를 확인합니다.

```bash
agent mgr status EPIC-100 --verbose
```

**상황**: `TASK-101(API 연동)`이 `TASK-102(DB 스키마 변경)`에 의해 막혀(Blocked) 있음을 발견합니다. 
`TASK-102`가 지연되면서 전체 일정이 밀리고 있습니다.

**조치**: 매니저는 `TASK-102`의 담당자와 대화 후, 작업의 일부를 쪼개어 선행 필수 사항만 먼저 처리하기로 결정합니다.

## Step 2: 작업 분할 및 히스토리 연결 (Split)
너무 큰 `TASK-102`를 두 개로 나눕니다.

1. `TASK-102` (원본): 필수 스키마 변경 (이번 스프린트)
2. `TASK-103` (신규): 추가 최적화 작업 (다음 스프린트)

```bash
# Jira에서 새 티켓 생성 및 Split 링크 연결
# (Jira UI 혹은 CLI를 통해 실행)
pm jira link create TASK-102 TASK-103 "Split"
```

이를 통해 원래 작업이 왜 쪼개졌는지, 어떤 티켓이 다음으로 이월되었는지 히스토리를 남깁니다.

## Step 3: 근본 원인 분석 (Causes / Is caused by)
배포 후 발생한 버그 `BUG-200`이 지난주에 처리한 `TASK-099` 때문임을 알게 되었습니다.

```bash
# 버그와 원인 작업 연결
pm jira link create TASK-099 BUG-200 "Causes"
```

**매니저의 분석**:
"최근 3개월간 발생한 버그의 40%가 `TASK-XXX` 형태의 급한 핫픽스 작업에서 `Causes` 링크로 연결되어 있네. 급한 일정 때문에 테스트가 누락되는 게 근본 원인이니, 핫픽스 프로세스를 개선해야겠다."

## Step 4: 시각적 관리 (Visual 강조)
매니저는 팀 보드 설정에서 `is blocked by` 링크가 있는 카드를 빨간색으로 강조하도록 설정하여, 매일 아침 병목 현상을 1초 만에 파악합니다.

---

## 결론
이슈 연결(Issue Linking)은 단순한 기록을 넘어 다음과 같은 가치를 제공합니다:
1. **투명성**: 누가 누구를 기다리는지 명확히 보임.
2. **데이터 기반 분석**: 반복되는 버그의 원인을 통계로 파악 가능.
3. **유연성**: 큰 작업을 `Split`으로 관리하며 프로젝트 속도를 유지.
